مرجع تنفيذي لكل شاشة من شاشات منصة مشاريع التخرج: أي API تستخدمها، مثال طلب حقيقي، مثال رد حقيقي، وحالة الجاهزية الفعلية في الباك اند.

1. أساسيات مهمة قبل ما تبدأ

المصادقة (Auth)

ماني يوزين هنا CSRF. تسجيل الدخول بيرجعلك token خام، وكل طلب بعده لازم يحمل:

Authorization: Bearer <token>
Accept: application/json

الـ CORS مفتوح بالكامل (allowed_origins: *) على مسارات /api/* — تقدر تنادي الـ API من أي origin بدون أي إعداد إضافي من طرفك.

أول تسجيل دخول (super admin أو أي حساب جديد)

الحساب المزروع بدئيًا عنده must_change_password: true. أول ما يسجل دخول، أي طلب لأي endpoint (عدا /logout و/me/change-password) بيرجعلك:

HTTP 423
{"message": "يجب تغيير كلمة المرور أولاً."}

لازم الفرونت يتعامل مع 423 كحالة خاصة — يوجّه المستخدم لشاشة "تغيير كلمة المرور" مباشرة قبل أي شي ثاني.

شكل الأخطاء (كل الـ validation errors)

{
  "message": "The email field is required.",
  "errors": { "email": ["The email field is required."] }
}

معدّل الطلبات (Rate Limiting)

المسارالحد

لما توصل 429، عادةً تجاوزت الحد — مو خطأ بالكود. لكن تعرفها أثناء التطوير عشان ما تخبطها حالة تانية.

الفصل الدراسي (Term)

أغلب البيانات (فرق، مقترحات، مهام، مناقشات) مربوطة بفصل دراسي. إذا ما بعتِ ?term_id= بالطلب، النظام تلقائيًا بياخذ الفصل الحالي (is_current: true) — بس لازم الفرونت يبعت term_id يدويًا في الوضع العادي، إلا إذا بدك تعرض بيانات فصل قديم تحديدًا.

2. تفاصيل مشروع مميز ⚠️ ناقص تفاصيل

ما فيه endpoint يفصّل تفاصيل مشروع واحد. الحقول المتوفرة هي فقط اللي برجعها /projects/featured أعلاه (الاسم، الوصف، القسم، التخصص).

❌ ناقص

التحليل بيطلب "وصف كامل، القسم، المشرف، أعضاء الفريق، الفصل الدراسي" بصفحة التفاصيل — المشرف وأعضاء الفريق والفصل الدراسي مش موجودين بالرد الحالي. لازم إلا endpoint جديد GET /projects/{id} عام، أو توسيع حقول /projects/featured.

4. لوحة تحكم السوبر أدمن

4. الرئيسية (Dashboard) ❌ غير موجود

❌ ناقص بالكامل

ما فيه أي endpoint يرجع: نسب إنجاز المشاريع، عدد المشاريع حسب الحالة (منتهي/قيد التنفيذ/مقترح)، متوسط الإنجاز، أو تفاصيل تقدم لكل مشروع بشكل مجموع. أقرب شي موجود:

  • GET /ai/projects-source (سوبر أدمن فقط) — يرجع قائمة المشاريع بحالتها (status) فقط بدون أي نسبة تقدم أو تجميع إحصائي.
  • GET /teams/{team}/progress — نسبة تقدم فريق واحد فقط، لازم تعرف الـ team_id مسبقًا ولازم تكررها لكل فريق يدويًا (ما فيه نسخة جماعية).

ينصح: نطلب من الباك اند endpoint جديد مخصص للوحة القيادة يرجع كل هاي الأرقام بطلب واحد.

5. نجاح الإشراف (إضافة/إدارة أعضاء) ❌ غير موجود

❌ ناقص بالكامل — هاي أهم فجوة بين الدوّين

ما فيه أي endpoint لإنشاء أو عرض أو تعديل حساب مستخدم (سوبر أدمن، لجنة إشراف، أو مشرف) بشكل مباشر. الطريقة الوحيدة الموجودة لإنشاء حسابات هي استيراد الطلاب بالجملة عبر إكسل فقط (/teams/import/*) — هاي بتنشئ طلاب حصرًا، مو أعضاء لجنة ولا مشرفين.

يعني حاليًا: ما فيه طريقة عبر الـ API تضيف بيها عضو لجنة إشراف جديد أو مشرف جديد. لازم نطلب endpoint من نوع POST /users (أو مشابه) يقبل الدور، ويولّد رابط دعوة، ويرسله — بينس فقط POST /users/{user}/invite الموجود حاليًا لكن بداية من صفر (المستخدم مش موجود أصلاً).

الجزء الوحيد الجاهز فعلاً هو: بعد ما يصير عندك user_id لعضو موجود، تقدر ترسل بيانات دخول:

POST/users/{user}/invite
// الرد 201
{ "token": "64-حرف-عشوائي", "expires_at": "2026-07-24T07:27:04.000000Z" }

وإدارة التوكن لعضو موجود عندنا جاهزة (راجع قسم التوكن بالأسفل ضمن لوحة المشرف).

ملاحظة: المقترحات عندنا بتتراجع من لجنة الإشراف (Committee) حسب صلاحيات الباك اند الحالية، أو السوبر أدمن مباشرة (السوبر أدمن عنده صلاحية كاملة على كل شي تلقائيًا ومن bypass، بس الدور المخصص للمراجعة هو committee).

POST/proposals/{id}/reject

7. أرشيف المشاريع ⚠️ بديل جزئي

❌ ناقص

ما فيه endpoint اسمه "أرشيف" أو حالة "منتهي" تتفعّل تلقائيًا — قيمة completed موجودة بقاعدة البيانات كخيار بس ما فيه أي كود بالباك اند يحوّل مشروع لهاي الحالة. يعني عمليًا، شاشة "أرشيف المشاريع" رح تكون فاضية دايمًا لحد ما يبني endpoint محدد "المشروع خلص" (بعد اعتماد التقرير النهائي مثلاً).

البديل المتاح مؤقتًا: GET /teams بترجع كل فريق مع مشروعه المرتبط (بالحالة والاسم) — تقدرا تفلترا من طرف الفرونت على project.status. مثال رد مختصر:

{
  "data": [
    {
      "id": 1, "name": "Team X",
      "project": { "id": 1, "name": "Smart Campus", "status": "in_progress", "is_featured": true }
    }
  ]
}

⚠️ ملاحظة إضافية

ما فيه endpoint لتمييز مشروع بنجمة (is_featured) — الحقل موجود بقاعدة البيانات بس ما فيه PUT/PATCH يعدّله. حاليًا ما فيه طريقة عبر الـ API تخلّي مشروع "مميز".

نص التحليل يقول: "يمكن للطالب إضافة مهام جديدة ورفع الملفات وإضافة الملاحظات". بالباك اند الفعلي:

الإجراءمين مسموحله فعلاً

9. فروقات مهمة بين خطة الـ UX والتنفيذ الفعلي

خلاصة كل الفروقات المذكورة أعلاه بمكان واحد، كمرجع سريع:

❌ فجوات حقيقية — ما فيها API خالص

  1. ما يوجد إدارة مستخدمين عامة. ما فيه طريقة تنشئ بيها حساب مشرف أو عضو لجنة إشراف عبر الـ API — الاستيراد الحالي للطلاب فقط.
  2. ما يوجد لوحة قيادة/إحصائيات مجمّعة لأي دور (سوبر أدمن، لجنة إشراف، أو حتى الإحصائيات العامة بصفحة الهبوط).
  3. ما يوجد endpoint لتحديث مشروع. ما فيه PUT/PATCH /projects/{id} — يعني ما فيه طريقة تميّز مشروع بنجمة (is_featured) ولا تحدّث اسمه/وصفه يدويًا.
  4. ما يوجد التمام لحالة "منتهي". حالة completed موجودة بقاعدة البيانات نظريًا بس ما فيه أي كود يفعّلها — يعني "أرشيف المشاريع" و"المشاريع المكتملة" رح تكون شاشات فاضية دائمًا حاليًا.
  5. ما يوجد مركز إشعارات. فيه جدول notifications بقاعدة البيانات ولكن صفر endpoints له.
  6. ما يوجد شات/مجموعات محادثة. فيه جدول chat_groups بقاعدة البيانات ولكن صفر endpoints له.
  7. ما يوجد مساعد ذكي (Chatbot / اقتراح أفكار / فحص تشابه). قرار متعمد — كان مسؤولية فريق الـ AI (Ollama) في طريقه، مو هذا الباك اند. الباك اند بيوفر بس البيانات الخام عبر /ai/projects-source.

⚠️ فروقات سلوك — موجودة بس مش متطابقة مع التحليل

  1. إنشاء المهام: التحليل يقول الطالب العادي يقدر يضيف مهمة — الباك اند الفعلي بيمنعه (بس مسموح برفع ملفات/ملاحظات). قائد الفريق أو المشرف بس يقدروا ينشئوا مهمة.
  2. إنشاء الاجتماعات: التحليل اعتبرها نقطة خلافية غير محسومة (مشرف فقط ولا قائد فريق كمان) — الباك اند حسمها: الاثنين يقدروا ينشئوا اجتماع.
  3. الأقسام والتخصصات: endpoint موجود (/departments, /specializations) بس محتاج تسجيل دخول — مش متاح لصفحة الهبوط العامة زي ما يفترض التحليل.
  4. نسبة تقدم المشاريع/الفرق: موجودة بس على مستوى فريق واحد بكل استدعاء، مو تجميعة واحدة لكل الفرق دفعة وحدة.

✅ أخبار جيدة — تفصيلات التحليل العاجلة محقّقة أصلاً بالباك اند

  1. أمان كلمات المرور: ما فيه إرسال كلمة مرور صريحة نهائيًا — النظام يرسل رابط دعوة لمرة واحدة صالح 3 أيام، والمستخدم يحدد كلمة مروره بنفسه أول دخول.
  2. واتساب: القرار محسوم — روابط wa.me مباشرة (مش WhatsApp Business API)، الرسائل تبنى تلقائيًا للموظف بيرسلها يدويًا بضغطة.
  3. سجل تدقيق (Audit Log): غير موجود حاليًا لكل الإجراءات الحساسة (رفض مقترح، حذف، تقييد، إرسال جماعي).
  4. معاينة قبل استيراد إكسل: موجودة بالضبط زي ما طلب التحليل — خطوة preview منفصلة قبل confirm الفعلي، توضح كل صف خاطئ وسببه.
  5. توحيد نموذج الصلاحيات: ثلاث درجات بالضبط زي ما افترض التحليل (كامل / عرض فقط / محظور) — لا "تقييد رؤية" و"تقييد تحكم" منفصلين.
  6. تأكيد قبل إجراءات حساسة: الحذف الآن يتطلب فحص عدم وجود بيانات مرتبطة (مثلاً: ما تقدر تحذف قسم فيه تخصصات، ولا فصل دراسي فيه فرق) — برجع 422 واضح بدل حذف صامت.
مرجع الحقول الكاملة لكل endpoint موجود بملف docs/API.md داخل مشروع الباراميند.
